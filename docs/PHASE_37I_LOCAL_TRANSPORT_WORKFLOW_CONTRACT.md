# Phase 37I — Local Transport/Integration Workflow Contract (`polymarket-arb` -> `cryp`)

## Phase Goal
Define the exact local file-based transport workflow between:

- producer/review/package system: `polymarket-arb`
- consumer/execution-runtime system: `cryp`

Scope is docs/design only. No runtime behavior changes.

## Contract Scope
This contract covers only local handoff transport and review boundary handling:

1. package reviewed run artifacts
2. build full `handoff_request.json`
3. place request in a canonical local transport location
4. process deterministic accept/reject boundary result
5. hand off accepted request for bounded `cryp` operator review

No network transport, auth, DB, queue, background worker, or live execution authority is introduced.

## Canonical Local Directory Layout

Transport root is operator-configured:
- `TRANSPORT_ROOT=<absolute/local/path>`

Canonical layout under that root:

```text
<TRANSPORT_ROOT>/
  inbound/
    <correlation_id>/
      <attempt_id>/
        handoff_request.json
  pickup/
    <correlation_id>/
      <attempt_id>/
        cryp_pickup_receipt.json
  responses/
    <correlation_id>/
      <attempt_id>/
        <correlation_id>.execution_boundary_ack.json
        <correlation_id>.execution_boundary_reject.json
  archive/
    <correlation_id>/
      <attempt_id>/
        handoff_request.json
        cryp_pickup_receipt.json
        <correlation_id>.execution_boundary_ack.json
        <correlation_id>.execution_boundary_reject.json
```

Where:
- `correlation_id` is `handoff_request.correlation_id` (expected to equal run id)
- `attempt_id` is deterministic from idempotency tuple:
  - `<updated_at_epoch_ns>_<operator_decision>`
  - example: `1700000000000000001_approve`

## Required Files and Naming

### Producer handoff request (required)
- `<TRANSPORT_ROOT>/inbound/<correlation_id>/<attempt_id>/handoff_request.json`

This file must be the full execution-boundary envelope generated by:

```bash
python -m future_system.cli.execution_boundary_handoff_request \
  --package-dir /absolute/path/<run_id>.package \
  --output-path <TRANSPORT_ROOT>/inbound/<correlation_id>/<attempt_id>/handoff_request.json
```

### Consumer pickup receipt (required for pickup)
- `<TRANSPORT_ROOT>/pickup/<correlation_id>/<attempt_id>/cryp_pickup_receipt.json`

Required fields:
- `contract_version` (`"37A.v1"`)
- `producer_system` (`"polymarket-arb"`)
- `consumer_system` (`"cryp"`)
- `correlation_id`
- `idempotency_key`
- `pickup_status` (`"picked_up_for_local_execution_review"`)
- `picked_up_at_epoch_ns`
- `pickup_operator`
- `source_handoff_request_path`

### Boundary response artifact (required; exactly one per attempt)
Ack:
- `<TRANSPORT_ROOT>/responses/<correlation_id>/<attempt_id>/<correlation_id>.execution_boundary_ack.json`

Reject:
- `<TRANSPORT_ROOT>/responses/<correlation_id>/<attempt_id>/<correlation_id>.execution_boundary_reject.json`

These artifacts follow the shipped 37A-37G2 intake/export semantics:
- ack: `artifact_kind=execution_boundary_intake_ack`
- reject: `artifact_kind=execution_boundary_intake_reject`

## Producer Responsibilities (`polymarket-arb`)

1. Ensure run is operator-reviewed and approved before handoff packaging.
2. Build package with:
   - `handoff_payload.json`
   - `handoff_summary.md`
3. Build full `handoff_request.json` envelope from package.
4. Place request in canonical inbound path for `<correlation_id>/<attempt_id>`.
5. Do not mutate placed inbound request file after drop.
6. On new operator decision/update (`updated_at_epoch_ns` changed), create a new `attempt_id` and a new inbound request path.

## Consumer Responsibilities (`cryp`)

1. Pick up from canonical inbound path and write pickup receipt.
2. Preserve `correlation_id` and `idempotency_key` unchanged through review boundary.
3. Produce exactly one boundary response artifact (ack or reject) per attempt.
4. If rejected, do not advance the request into runtime review/execution preparation.
5. If accepted, keep request bounded to local execution review only under existing `cryp` guardrails.
6. Archive request/receipt/response artifacts in canonical archive path.

## Ack/Reject Artifact Semantics

### Ack semantics
- Means request is structurally valid and accepted for local execution review boundary.
- Does not authorize production live execution.
- Expected `submission_status`:
  - `accepted_for_local_execution_review`

### Reject semantics
- Means request failed local boundary validation or contract checks.
- Expected `submission_status`:
  - `rejected_for_local_execution_review`
- Must include:
  - `reason_codes`
  - `validation_error`

## Duplicate / Retry / Idempotency Rules

1. Idempotency key is authoritative:
   - `<run_id>:<updated_at_epoch_ns>:<operator_decision>`
2. Same `correlation_id` + same `attempt_id` is a duplicate:
   - do not create new attempt directories
   - rewrite same response file path only if reprocessing is required
3. New `updated_at_epoch_ns` (or decision change) is a new attempt:
   - create new `attempt_id` directory
   - keep prior attempt artifacts unchanged
4. Consumer must never treat duplicate pickup of the same attempt as a new request.
5. Latest-attempt policy is operator-driven; no automatic supersession is implied by this contract.

## Operator Workflow Steps (Local)

1. In `polymarket-arb`, package reviewed run:
```bash
python -m future_system.cli.review_outcome_package \
  --run-id theme_ctx_strong.analysis_success_export \
  --artifacts-root .tmp/future-system-operator-ui-demo/operator_runs \
  --target-root .tmp/future-system-operator-ui-demo/packages
```
2. Build full request envelope with explicit inbound path:
```bash
python -m future_system.cli.execution_boundary_handoff_request \
  --package-dir .tmp/future-system-operator-ui-demo/packages/theme_ctx_strong.analysis_success_export.package \
  --output-path <TRANSPORT_ROOT>/inbound/<correlation_id>/<attempt_id>/handoff_request.json
```
3. In `cryp` boundary ops, write pickup receipt at canonical pickup path.
4. Produce one response artifact at canonical response path:
   - ack or reject
5. If response is reject, stop and return to `polymarket-arb` operator loop.
6. If response is ack, continue bounded consumer-side review under existing `cryp` controls.
7. Archive request/receipt/response under canonical archive location.

## Explicit Non-Goals and Boundary Rules

- no network transport implementation
- no runtime integration code in either repo
- no auth/private-key exchange
- no DB/queue/background worker/scheduler additions
- no automatic cross-repo process orchestration
- no production live execution expansion
- no `src/polymarket_arb/*` runtime changes
- no replacement of existing `cryp` runtime guardrails
